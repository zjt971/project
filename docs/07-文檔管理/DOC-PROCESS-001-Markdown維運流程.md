# RI-3-2 Markdown 維運與通知流程

> 任務說明：定義以 Markdown 為基礎的版本控管與通知流程，取代 Jira/Confluence，支援專案協作。

---

## 1. 版控流程

1. **儲存庫維護**
   - 所有 Markdown 檔案皆存於 Git 儲存庫 `RI/docs/` 內。
   - 採用 `main` 分支為事實來源，如需重大調整可使用功能分支（`feature/<任務編號>`）。

2. **提交原則**
   - Commit 訊息格式：`[RI-<Story>-<子任務>] <摘要>`。
   - 同一提交應聚焦單一主題；若同時間更新多個檔案，需在提交說明中列出重點。
   - 為保持易讀性，建議每個 Markdown 檔案減少大量無關變更（避免噪音）。

3. **審核與合併**
   - 重大內容（Roadmap、Milestone 調整）建議發送 Merge Request 由 PO/PMO 審閱。
   - 例行紀錄（會議紀錄、任務進度）可直接由 Scrum Master 或負責人提交並於每日站會通報。

---

## 2. 檔案更新通知

- **即時通知**：更新關鍵檔案（專案計畫、Scrum/Sprint 計畫、任務追蹤、Action Items）時，於 Teams/Slack 的 `#reinsurance-project` 頻道發布訊息。
  - 格式建議：
    ```
    [更新通知] RI-1-1 會議紀錄已上傳
    - 檔案：../01-專案管理/PM-MEETING-001-啟動會議紀錄.md
    - 重點：新增決議與行動項目（A01-A08）
    - 如需回饋請於 9/29 前提供
    ```
- **週報同步**：Scrum Master 每週五整合本週主要檔案更新，於站會及郵件週報中提醒團隊檢閱。
- **月度匯總**：PMO 於 Steering Review 前彙整當月重大更新，確保決策層了解最新狀態。
- **規範更新**：任何涉及開發規範、ADR、UI 導則的變更，都必須在通知訊息中標註「#Guardrails」，並附上連結與簡述內容調整，提醒開發者檢閱。

---

## 3. 任務追蹤表同步

- 任何檔案新增/更新完成後，負責人需在 `../01-專案管理/PM-REPORT-001-任務追蹤.md` 中：
  - 勾選對應任務狀態。
  - 於必要時新增連結或備註（例：指向新產出物、提醒審閱截止日）。
- Scrum Master 每週檢閱一次追蹤表，確認狀態與文件內容一致。

---

## 4. 回饋與修訂流程

- 建議使用 Pull Request 或留言進行內容審查；若無法使用工具，可在 Action Item 表新增待回饋項。
- 需修訂的檔案，請於原檔案底部更新「版本紀錄」段落，記錄日期、版本、說明、作者。
- 重大回饋（例如需求變更）須同步通知 PO/PMO，並更新專案/產品計畫文件。
- 若變更涉及工程守則（資料庫、程式架構、UI 規範），請建立或更新對應的 Task Tracker 項目，指派相關人員確認所有模組已依新規調整。

---

## 5. 備援與恢復

- Git 儲存庫每日備份至遠端伺服器（由 DevOps 設定）。
- 若發生誤刪或衝突，可透過 Git 歷史版本還原。
- 建議每月底匯出主要文件（Project Plan、Scrum Plan、Task Tracker）成 PDF/HTML 作為快照備查。
